# 成交量异动策略优化方案 —— 评估与评审意见

> 本文档是对 `docs/volume_anomaly_optimization_plan.md` 的逐项评审。
> 评审方法：对照当前 `volume_anomaly` 插件的实际实现（`data.py` / `prompts.py` / `schemas.py` / `pipeline.py`），逐条评估方案的合理性、与现状的边际价值、实施代价、潜在风险，并按 ROI 给出优先级排序与实施波次建议。

---

## TL;DR — 总体结论

原方案**方向正确、维度全面**，作者对 A 股主升浪逻辑（VCP / O'Neil / 资金流 / 板块共振）和 AI 量化的演进路径（Hybrid Models / Multi-Agent / RAG）都有清晰的把握。但作为一份"实施驱动"的方案，它有三个需要补强的地方：

1. **缺少优先级和 ROI 排序**：所有改进项并列出现，未区分"低成本快速验证"与"高成本战略投资"。
2. **缺少可证伪性指标**：每条优化的"成功标准"没有定义（比如增加 RPS 后 hit rate / 主升浪命中率应提升多少才算有效）。
3. **缺少前置基础设施识别**：第 4 节的 RAG / Multi-Agent / LightGBM 路径都依赖一个共同的、当前缺失的能力——**T+N 自动回测与标注闭环**。这个前置能力没在方案中被点出。

我的整体倾向是：**优先做能立刻提升信号质量的"低成本特征/Prompt 改进"，同时尽快搭建 T+N 回测闭环作为后续所有 ML/RAG 演进的基础设施**。把"多智能体辩论 / LightGBM 前置打分 / Post-Mortem RAG"这些高级架构往后排——它们在没有数据沉淀的情况下投入产出比并不高。

---

## 1. 筛选条件优化（ScreenRules）

### 1.1 动态换手率阈值

**方案核心**：把 `turnover_min/max = [3.0, 10.0]` 改为基于流通市值的分桶阈值，或基于自身 60 日分位数的相对阈值。

**评估**：

- **"按流通市值分桶"——强烈认同**。当前固定区间 [3, 10] 对小盘股偏向严重：千亿市值股 5% 已是天量，但小微盘 10% 只是日常活跃。这是当前规则对"大盘真异动"漏报、对"小盘伪活跃"误报的根因之一。
- **"自身 60 日分位数"——价值有限，与现有规则同质**。当前 `vol_ratio_5d_min = 2.0` 加 `vol_t == max(short_window)` / `top_N(long_window)` 已经在做"今天是否比自身历史活跃"的判断。换手率分位数本质是同一个"自比较"逻辑的换皮。新增一个同向信号，对 hit rate 的边际提升不大。
- **代码侧**：`daily_basic` 已经在 screen 阶段被调用了（`turnover_rate` 已到手），只需要再读一次 `circ_mv` 字段（同一个 API 已返回），分桶阈值表写在 `ScreenRules` 里即可。代价极低。

**优先级**：🟢 **高**（仅推进市值分桶，不做分位数）。
**建议实施**：在 `ScreenRules` 增加 `turnover_buckets: list[tuple[float_circ_mv_yi_max, float_min, float_max]]`，例如：

```python
# (流通市值上限_亿, turnover_min, turnover_max)
DEFAULT_TURNOVER_BUCKETS = [
    (50,    5.0, 15.0),   # 微盘
    (200,   3.5, 12.0),   # 中小盘
    (1000,  2.5,  9.0),   # 中盘
    (10**9, 1.5,  6.0),   # 大盘/超大盘
]
```

向后兼容：保留 `turnover_min/max` 作为"未启用分桶"时的 fallback。

### 1.2 形态过滤精细化（上影线控制）

**方案核心**：当前只控制 `body_ratio >= 0.6`、未限制上影线，建议增加 `upper_shadow_ratio < 0.3` 过滤"避雷针"形态；同时把涨幅放宽到 `[4.0, 9.5]`。

**评估**：

- **上影线过滤——强烈认同，是最 cheap 而 effective 的改进之一**。现有规则下，`body_ratio = 0.6` 允许 40% 的振幅落在影线，理论上可以是纯上影线（极端"冲高回落"），这正是放量异动里最危险的"主力试盘失败"形态。新增过滤几乎没有数据成本——所需的 `open / high / low / close` 都已在 `daily_t` 中。
- **0.3 这个阈值偏紧，建议起始 0.35–0.40**。A 股很多真实主升浪首日就是"高开冲高—盘中震荡—尾盘修复"，过严会漏掉伪装成"试盘"的真主力。建议把 `upper_shadow_ratio_max` 设为可调，默认 0.35。
- **涨幅放宽到 9.5——需要谨慎**。9.5 已经接近涨停（10%），逻辑上"差点涨停的强势股"应该让 `limit_up_board` 策略捕获，而 `volume_anomaly` 的目标是"未达涨停但已强烈放量"的标的。两个策略的边界要清晰，避免重叠采样。建议涨幅上限保持 8.5–9.0 之间，下限可适度放宽到 4.5。

**优先级**：🟢 **高**。改动 < 20 行代码，预期能显著降低"高位放量出货"误报。

**建议实施**：在 `daily_t` 计算阶段增加：
```python
daily_t["upper_shadow_ratio"] = (daily_t["high"] - daily_t[["open","close"]].max(axis=1)) / daily_t["range"]
```
然后在过滤链里加上 `& (daily_t["upper_shadow_ratio"] <= rules.upper_shadow_ratio_max)`。

### 1.3 多日量价堆积识别（连续温和放量）

**方案核心**：把"$V_T > V_{T-1} > V_{T-2}$ 且伴随价格抬升"这类"温和放量阶梯"作为 `vol_ratio_5d >= 2.0` 的并列通过条件（OR）。

**评估**：

- **理论上认同——这确实是另一类高质量的"主升浪前兆"信号**（量化研究里被反复验证）。
- **但落地路径需要小心**：
  - 若以"OR"形式合并到现有规则，**会把规则边界搞模糊**。一旦命中变多，难以分辨是"暴量逻辑"还是"温和堆积逻辑"贡献的，调试成本上升。
  - 该模式与"暴量异动"在交易心理上代表的是不同的主力行为（暴量 = 短期博弈/异动突破；温和堆积 = 长期吸筹/平台抬升），混在同一池里跑同一套 LLM Prompt 可能拖累后者的判断质量。
- **建议改为"独立模式"**：在 `ScreenRules` 增加 `screening_mode: Literal["burst", "stair_step", "both"]`，`stair_step` 模式有自己一套阈值（不要求 vol 暴量 2x，但要求 3 天 vol 单调递增 + 3 天 close 单调递增 + 当天 body_ratio + 涨幅在 2-6%）。在 LLM 阶段标记 `entry_mode` 字段，让 Prompt 区分判断标准。

**优先级**：🟡 **中**。先把 burst 模式调到稳态再扩展，避免规则边界模糊化。

## 2. 补充量化因子（喂给 LLM 的输入侧）

### 2.1 VCP / 波动率收缩（ATR / 布林带宽度）

**方案核心**：增加 60 日布林带宽度，或 10 日 ATR 在过去 60 天中的分位数。

**评估**：

- **强烈认同**。当前 `_build_candidate_row` 已经计算了 `base_max_drawdown_pct`（价格收敛维度）和 `base_vol_shrink_ratio`（成交量收敛维度），缺的正是**波动率收敛**这一第三维度。三维齐全才能完整刻画 VCP。
- **数据成本：零**——所有需要的字段（`high / low / close`）已经在 60 日 `daily_df` 中，纯本地计算。
- **建议实现层面**：
  - 计算 10 日 ATR 序列，给出 `atr_10d_quantile_in_60d` 字段（值越小代表波动越收敛）
  - 计算 20 日布林带宽度的当前值与 60 日均值之比，给出 `bbw_compression_ratio` 字段（< 1 代表收敛中）
  - 这两个标量加到 `_build_candidate_row` 输出里，对 token 占用增量微小

**优先级**：🟢 **高**。零数据成本、与现有特征互补、对 LLM 的 `washout_quality` 判定有直接补强。

---

### 2.2 RPS / 相对强弱

**方案核心**：提供个股相对万得全 A / 上证指数的 20 日 / 60 日 alpha。

**评估**：

- **强烈认同**——这是 O'Neil/CANSLIM 体系的核心因子，A 股实证也广泛验证"启动前抗跌"的有效性。
- **实施层面注意点**：
  - 需要新增 Tushare API 调用：`index_daily(ts_code='000001.SH'/'000300.SH', ...)`，并扩到 60 日窗口。注意当前 `tushare_client` 的 cache 语义（trade_day_immutable）可以复用，第二次调用零成本。
  - 万得全 A（`881001.WI`）需要 Wind 数据，Tushare 通常用 `000300.SH`（沪深 300）或 `000985.CSI`（中证全指）作为 baseline 更合适。
  - 计算应该用对数收益率（避免高低水位非对称），输出 `alpha_20d_pct` / `alpha_60d_pct` / `rps_60d_rank_pct`（如果能拿到全市场的话——但全市场 RPS 计算成本较高，建议先做 alpha 即可）。
- **风险**：单一指数 baseline 在板块行情中可能误判（比如新能源板块普涨时，所有板块内成员的 alpha 都很高，区分度下降）。**改进版**：同时提供个股相对**所属行业**的 alpha——这比相对大盘 alpha 信号更纯。当前 `industry` 字段已在 watchlist 里，可以做到。

**优先级**：🟢 **高**。差异化信号，落地成本可控。

---

### 2.3 关键阻力位距离（120/250 日）

**方案核心**：计算当前收盘价距 120 日 / 250 日最高点的百分比，帮助 LLM 区分"突破前高放量"与"下跌通道反弹放量"。

**评估**：

- **认同——这是当前模型最缺的一块**。`high_60d` 在很多场景里就是异动当天本身（异动当天创了 60 日新高），让该字段失去区分能力。扩到 120/250 日才能真正承担"判断当前价位在历史中的位置"的功能。
- **实施代价**：
  - 简单做法：扩大 `_fetch_daily_history_by_date` 的窗口到 250 日。但这会把 LLM 输入 token 也撑大（如果 verbatim 喂数据）——必须在 `_build_candidate_row` 里聚合后只输出标量。
  - 更优做法：保留现有的 60 日 verbatim 窗口（用于 LLM 形态判断），单独跑一个 250 日的"轻量窗口"只为了算极值。这样 token 不增长，只多一次 API 调用（且 cache 友好）。
- **建议输出字段**：
  - `dist_to_120d_high_pct`（当前 close 距 120d 最高的下浮 %，0 表示创新高）
  - `dist_to_250d_high_pct`（同上，年线视角）
  - `is_above_250d_high`（bool，强势新高标记）

**优先级**：🟢 **高**。这是真正能让 LLM 准确判断 `pattern: breakout` 的关键信号。

## 3. LLM 提示词与推理优化

### 3.1 假突破（Bull Trap）风险评估

**方案核心**：在判断维度中强制要求 LLM 评估"诱多出货"概率，输出 `High Bull Trap Risk` 标签并给 `not_yet`。

**评估**：

- **方向认同，但实施细节需要调整**。当前 Prompt 的维度 F (风险) 已经覆盖了"高位放量出货 / 流通盘过大 / 题材孤立"等，但并没有"必须强制评估"的硬性要求，且 `risk_flags` 是字符串数组，模型可以自由发挥（甚至漏掉）。
- **强制化的副作用**：如果硬要求"必须输出 Bull Trap 评估"，Few-Shot 偏向稍微一变，可能让 LLM 对所有放量都套上 "中等 Bull Trap Risk" 标签——过度防御化，让 `imminent_launch` 数量骤降。
- **更稳的做法**：增加一个**结构化字段** `bull_trap_risk: Literal["low", "medium", "high"]`，让模型必须显式打分（而不是塞进自由文本 risk_flags）。然后在 Prompt 里给出明确判断标准（"long upper shadow + falling moneyflow + late-stage sector → high"）。结构化字段比自由文本更容易在后续做归因和 A/B 验证。

**优先级**：🟡 **中**。合理但需要小心调参，建议和 Few-Shot 一起上。

---

### 3.2 结构化思维链 / 显式分维度评分

**方案核心**：让 LLM 按 "(1) 洗盘深度 → (2) 形态突破 → (3) 资金共振 → (4) 结论" 的顺序思考。

**评估**：

- **方向认同，但有更好的形式**。当前 `rationale` 上限 200 字、`key_evidence` 1-5 条，已经是相当结构化的输出。再叠加"思维链顺序"的隐式要求，对一个已经被 schema 严格约束的模型，边际收益有限。
- **更有价值的演进**：从"隐式 CoT"升级到**显式分维度评分**。在 schema 里增加：

  ```python
  dimension_scores: {
      "washout":    int 0-100,
      "pattern":    int 0-100,
      "capital":    int 0-100,
      "sector":     int 0-100,
      "risk":       int 0-100,    # 越高代表风险越大
  }
  # launch_score = weighted_sum(...) - risk_score
  ```

  这样每个维度的判断都"上秤"，模型不能在维度间游移。**关键收益**：调试和归因时一眼能看出"是哪个维度拖低/拉高了 launch_score"，这对后续 prompt 迭代是巨大的能见度提升。也能直接接到下游做"按维度排序"的视图。

**优先级**：🟢 **高**（升级为显式分维度评分版本）。

---

### 3.3 交易视角的盈亏比预估（防守位 / 止损位）

**方案核心**：让 LLM 给出具体防守价格（如 `stop_loss_price: 15.30`），下游可直接挂单。

**评估**：

- **方向认同，但 LLM 直接输出价格是错的姿势**。LLM 在数字精度上有显著幻觉风险（小数点位、四舍五入、单位混淆），让它直接出"15.30"这种数值，非常容易翻车——尤其在低价股（< 5 元）和高价股（> 100 元）混合的批次里。
- **正确姿势**：让 LLM 选择**防守锚点类型**（结构化枚举），由本地代码计算实际价格。

  ```python
  defensive_anchor: Literal["MA10", "MA20", "MA60", "platform_low", "anomaly_open"]
  invalidation_threshold_pct: float  # 跌破锚点多少 % 视为破位
  ```

  下游在接到结果后，从 `_build_candidate_row` 已经计算的 `ma10/ma20/ma60` 直接取价，再乘以 `(1 - threshold_pct/100)`，得到精确止损价。**算 RR 比、做仓位管理、回测**都基于这个本地计算的值，不依赖 LLM 数字精度。

**优先级**：🟡 **中**（建议改成"锚点类型"形式，而不是直接价格）。

---

### 3.4 Few-Shot 示例对齐

**方案核心**：在 system prompt 里加 1-2 个极简 JSON 示例（一个 "教科书 VCP 突破"，一个 "高位巨量烂板"）。

**评估**：

- **强烈认同——这是 ROI 最高的单点改进之一**。
- 当前 prompt 只提供了 schema 规范、缺少"判断尺度"的 anchoring。同样的输入数据，DeepSeek / Claude / GPT 给出的 launch_score 可能差 20+ 分，根因就是缺少 calibration anchor。
- **代价**：system prompt 增加 ~600-1000 tokens（每批每次都消耗）。对于 200K 的输入预算来说，是可以接受的边际成本。
- **实施建议**：
  - 例 A：典型"VCP 充分洗盘 → 放量突破"模板，示范 `imminent_launch + sufficient + breakout` 配套的 evidence 与 rationale 写法
  - 例 B：典型"高位长上影线 + moneyflow falling + 题材末段"模板，示范 `not_yet + insufficient` 配套的写法
  - **关键**：两个例子里 `key_evidence` 必须严格用真实输入字段名（如 `base_max_drawdown_pct`、`net_mf_trend`），让模型对齐"用哪些字段、用什么单位"

**优先级**：🟢 **高**。

## 4. 全局 AI 量化架构与交易系统演进

> 这一节属于"战略路径"评估。我对方案的总体看法是：**方向都对，但优先级排错了**。下文逐项展开。

### 4.1 日内分时特征

**评估**：方向正确，但实施代价高。
- Tushare 的 `stk_mins` / `min` 接口需要更高的权限档位
- 数据量大（日 240 条 × 候选数）→ token 与 API 调用成本都上一个台阶
- **真实最大障碍**：盘后才能拿到完整分时，**与本策略"次日交易"的时序冲突**——除非把异动捕捉延后到收盘后的 18:00（已支持 `close_after = 18:00` 时序），且接受"次日开盘前才出预测"的延迟

**优先级**：🟡 **中**（属于"二期"）。先把日线维度的特征工程吃透，再考虑下沉到分时。

---

### 4.2 板块/题材动态图谱

**评估**：当前已经有 `sector_strength_data`（`limit_cpt_list` / `industry_fallback`），但确实是单日静态的。
- **简化版增强**：把 `limit_cpt_list` 拉近 3-5 日的，计算"连续上榜板块" → 主线标记。再把候选股所属题材与主线对照，在 LLM 输入中加 `is_in_main_theme: bool` 字段。
- 实施代价低（多几次 API 调用，cache 友好），收益边际明显。

**优先级**：🟢 **高**（改"动态图谱"是大词，但这个简化版增强 ROI 极佳）。

---

### 4.3 LightGBM + LLM 混合架构

**评估**：方向上是经典做法（"传统 ML 给打分，LLM 做形态把关"），但**前提条件不成熟**。
- **前置缺失**：当前没有标注数据集——需要先有 T+3/T+5 收益作为 label 才能训 LightGBM。当前 `va_anomaly_history` 表存了历史 hits，但**没有自动 fetch T+N 收益的闭环**。
- **基础设施代价**：训练 pipeline、特征 store、模型 serving 全栈都要搭。这是季度级工程量。
- **是否值得**：在样本量大、特征齐全的场景，LightGBM 比 LLM 更准（A 股因子领域已被验证），但需要至少 6-12 个月的 hits 沉淀。当前急于上 ML，反而是**用一个不准的小模型替代一个 OK 的大模型**。

**优先级**：🔴 **低**（不是因为方向错，而是前置条件不成熟）。**但前置基础设施 ——"T+N 自动回测/标注闭环"必须排到最高优先级**（详见下一节）。

---

### 4.4 多智能体辩论（红方 / 蓝方 / 决策者）

**评估**：研究界很热（Multi-Agent Debate 论文不少），但在本场景的 ROI 不高。
- **Token 成本翻 3 倍**（红方 + 蓝方 + 决策者各一次完整 candidate 推理）。日常运行成本上一个量级。
- **收益边际递减**：单 LLM 如果 prompt 设计够好（特别是上了 4.3.2 的"显式分维度评分" + 4.4 的 Few-Shot），已经能在 system prompt 里覆盖正反两面（"先列做多证据，再列风险证据，最后综合"）。多智能体相对单 LLM 的提升主要在边缘样本（launch_score 50-70 之间）。
- **真要做，建议更轻量的形式**：单 LLM 调用，但 schema 里强制输出 `bull_case: list[str]` 和 `bear_case: list[str]` 各 2-3 条，最后再综合到 launch_score。token 增量 ~30%，收益接近双 agent 的 80%。

**优先级**：🔴 **低**。先把单 LLM 调到天花板再考虑。

---

### 4.5 自我进化（Post-Mortem RAG）

**评估**：**战略上极正确，短期收益低，但前置必须做**。
- **前提**：必须先有"T+N 自动回测能力"。即对每只历史 `imminent_launch` 标的，自动 fetch T+1 ~ T+5 OHLCV，计算实际收益，与预测对比。这一步是**所有后续 ML/RAG 路径的基础**。
- **当前缺口**：`va_anomaly_history` 表只存预测，不存"事后真实收益"。
- **完整闭环路径**：
  1. 新增 `va_realized_returns` 表：`(ts_code, prediction_date, t1_close, t3_close, t5_close, max_close_5d, return_t3, return_t5, max_dd_5d)`
  2. 新增定期 job：每个交易日自动跑 N 日前的预测、写入实际收益
  3. 接 Lesson schema：失败案例 → 复盘 Agent → 向量库
  4. RAG 注入：当前预测时检索相似失败案例

- **价值**：6-12 个月数据沉淀后，构成本策略的核心护城河。
- **关键认知**：**4.3 / 4.4 / 4.5 都依赖步骤 1-2。把它做出来，等于一次性给所有后续高级架构铺平了路**。

**优先级**：
- **战略上 🟢 极高**（前置基础设施）
- **当期 🟡 中**（需要的工程量是中等的，但不会立即提升当期信号质量）

---

### 4.6 交易执行层数值化输出

见 3.3 评估，建议改为"锚点类型 + 本地计算价格"。

**优先级**：🟡 **中**（与 3.3 合并实施）。

## 与原方案的关键分歧（明确列出）

为了避免后续讨论时的隐含偏差，这里把我和原方案不一致的点显式列出：

1. **关于"自身换手率分位数"**：我认为它与现有的 `vol_ratio_5d` 信号高度同质，不建议加。改为**按流通市值分桶**才能解决"小盘偏向"的真问题。

2. **关于涨幅放宽到 9.5**：我担心与 `limit_up_board` 策略的边界重叠，建议上限保持 8.5–9.0。

3. **关于多日量价堆积"作为 OR 条件"**：我认为应该作为**独立筛选模式**，而不是简单 OR，否则规则可解释性下降。

4. **关于 LLM 直接输出止损价格**：LLM 数值幻觉风险高，建议改为"防守锚点类型 + 本地计算价格"。

5. **关于"结构化思维链"**：我建议**升级为显式分维度评分**（而不是隐式 CoT 顺序），可见度收益更大。

6. **关于多智能体辩论**：在当前阶段 token 成本×3 的代价，相对单 LLM + 显式分维度评分的边际收益不大，应排到最后。

7. **关于 RAG / LightGBM**：方向正确，但**前置依赖一个原方案没点出的基础设施 —— T+N 自动回测/标注闭环**。这个前置不做，所有 ML/RAG 路径都是空中楼阁。

## 一个被原方案忽略的重要维度：可证伪性

每条优化项落地时，应该明确**"如何验证它有效"**：

- 上影线过滤：改造前后，`imminent_launch` 命中率（T+3 收益 > 0）应有 X% 的提升
- VCP 因子：在 LLM 输出中，命中 `washout_quality = sufficient` 的样本，T+3 平均收益应高于 `partial`
- RPS：把候选按 RPS 分位数分组，高分位组的命中率应单调高于低分位组

**没有这套验证框架，每条优化都是信仰投入**。这恰好也是 4.5（T+N 自动回测闭环）必须先做的另一个理由——它同时是"演进基础设施"和"实验验证基础设施"。

## 最终建议

1. **先做波次 1（5 项 P0）**——可控、低风险、收益立竿见影。
2. **同步启动波次 2 的 T+N 自动回测闭环**——这是后续所有演进的底盘，越早开始越早收效。
3. **暂缓波次 3-4 的高级架构**，等基础打牢、数据沉淀够了再上。

如果按这个节奏走，**3-6 个月**后该策略会从"严谨的量化指标读取器 + 单 LLM 评估"升级为"特征丰富的量化筛选器 + 可归因可调优的 LLM 评估系统"，并具备启动 ML/RAG 演进的全部前置条件。这条路径比"一上来就上 Multi-Agent + RAG"现实得多，也稳得多。
